# InsightFlow 面试表达说明

## 1. 为什么做这个项目

我选择做 InsightFlow，是因为智能问数是 AI 数据产品里非常典型、也非常容易落地的场景。

在真实业务中，运营、城市经理、业务负责人经常会问“为什么 GMV 下滑”“哪个区域订单上涨”“用户数变化如何”这类问题。传统链路需要分析师理解问题、写 SQL、查数、做归因，再输出解释。InsightFlow 的目标是把这条链路拆成一个可运行的 AI workflow 原型。

## 2. 项目解决什么问题

这个项目解决的是自然语言业务问题到数据分析结果之间的转换问题。

它重点解决三类问题：

1. 业务语言不结构化，系统需要把它转成指标、区域、时间和任务类型。
2. 数据分析链路长，需要自动生成 SQL 并执行确定性计算。
3. AI 输出需要有评测机制，不能只看回答是否“像那么回事”。

## 3. Workflow 如何设计

我把 workflow 拆成五步：

1. Intent Parsing：识别 metric、city、district、task、time_range。
2. SQL Generation：根据结构化 intent 生成取数 SQL。
3. Data Analysis：用 pandas 计算本周、上周、环比变化和候选影响因素。
4. Business Diagnosis：把计算结果组织成业务可读的诊断文案。
5. Evaluation：通过 benchmark 检查 parser 的准确率。

这个拆法的好处是边界清晰。模型或规则负责理解和生成，数据计算交给 SQL 或 pandas，评测模块负责验证输出质量。

## 4. Prompt 如何调优

当前 demo 没有接真实大模型，而是用规则模拟 prompt / parser 的输出。

如果接入 LLM，我会按三个版本迭代 prompt：

1. V1 基础版：只要求模型提取指标、地区和时间。
2. V2 Schema 约束版：告诉模型只能从已有字段中选择，降低字段幻觉。
3. V3 Few-shot 示例版：加入典型中文业务问题和标准 JSON 输出示例。

调优不会只看主观体验，而是用 benchmark 对比 metric、district、task 和整体准确率。

## 5. 如何评估效果

我会从两层评估：

第一层是模块评测：

1. Query 解析准确率
2. SQL 字段和筛选条件正确率
3. 输出格式正确率
4. 诊断结果是否由数据支撑

第二层是产品效果：

1. 用户从提问到拿到分析结果的耗时
2. 业务用户是否采纳诊断结果
3. 人工审核通过率
4. 错误归因和幻觉率

当前项目已经用 pytest 覆盖 parser、SQL、analytics、evaluator 和主流程 smoke test。

## 6. AI 能力边界如何控制

这个项目里我刻意把 AI 能力和确定性计算分开：

1. AI 不直接计算 GMV、环比和主因，计算交给 pandas 或数据库。
2. SQL 生成后需要做字段、表名和条件校验。
3. 业务诊断只能基于数据变化，不能凭空给出因果结论。
4. 对高风险分析结果，应保留人工审核和解释追溯。
5. 如果后续接真实大模型，需要增加输出 JSON schema 校验、SQL 安全校验和答案一致性评测。

这个边界设计可以降低幻觉风险，也更符合企业内部数据产品的落地方式。
