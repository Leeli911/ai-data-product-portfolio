# InsightFlow 项目介绍稿

## 项目出发点

我做 InsightFlow，是因为智能问数是 AI 数据产品里很典型的入口场景。

在业务现场，运营或城市经理不会一上来就说“请帮我查询某张表里的某个字段”，他们更常见的表达是：“为什么北京朝阳区本周 GMV 下滑？”“海淀区订单为什么上涨？”这类问题背后其实包含了指标、区域、时间、任务类型和归因分析诉求。

所以这个项目不是单纯做一个问答页面，而是想把自然语言问题到数据分析结论之间的链路拆出来，做成一个可以运行、可以评测、可以继续迭代的 demo。

## 我希望它解决的问题

InsightFlow 主要解决的是“业务问题如何进入数据分析系统”的问题。

我把它理解成三层：

1. 第一层是理解问题：用户说的是 GMV、订单、用户数，还是客单价。
2. 第二层是生成取数逻辑：把自然语言里的城市、区域、时间范围转成 SQL 条件。
3. 第三层是解释结果：不是只返回数字，而是告诉用户变化幅度和主要影响因素。

这个链路如果完全交给模型黑盒处理，短期看起来很快，但很难保证稳定性。因此我在项目里把解析、SQL、计算和诊断拆成了独立模块。

## Workflow 设计

当前 workflow 是这样设计的：

```text
用户输入中文问题
  -> parser 解析结构化 intent
  -> sql_generator 生成 SQL
  -> analytics 读取 mock 数据并计算环比
  -> diagnosis 输出业务解释
  -> evaluator 做 parser benchmark
```

这里我刻意让 pandas 承担确定性计算，而不是让 AI 直接计算 GMV 或环比。这样做的好处是，模型可以专注在理解和表达，关键指标计算仍然由可复现的程序完成。

## Prompt 和解析策略

当前版本没有接真实 LLM，我先用规则模拟 prompt 解析结果。

这样做不是为了回避模型能力，而是因为在做 AI 数据产品原型时，我希望先把字段、任务类型、输出格式和评测标准定下来。等这些边界稳定后，再接入真实 LLM 会更清楚应该评估什么、优化什么。

如果后续接入模型，我会从三个方向迭代：

1. 增加 schema 约束，让模型只能从已有字段中选择。
2. 增加 few-shot 示例，让模型学习中文经营问题到 JSON intent 的映射。
3. 增加输出校验，避免模型生成不存在的字段、区域或任务类型。

## 效果评估

这个项目里我没有只停留在“页面能跑”，而是加了 benchmark 和 pytest。

我关注的评估点包括：

1. parser 是否正确识别 metric、district、task。
2. SQL 是否包含正确字段、表名和筛选条件。
3. analytics 的环比计算和主因判断是否能被测试复现。
4. 主流程是否能从中文问题稳定跑到业务诊断输出。

这些测试不复杂，但它们能说明一个产品思路：AI 数据产品要有评测闭环，不能只依赖主观感受判断效果。

## 能力边界

我在项目里保留了几个明确边界：

1. 当前 demo 不连接真实数据库，SQL 用于展示取数逻辑。
2. 当前 demo 不调用真实大模型，规则解析用于模拟 LLM workflow。
3. 业务诊断只基于 mock 数据中的指标变化，不做超出数据证据的因果判断。
4. 如果接入真实模型，需要增加 SQL 安全校验、JSON schema 校验和人工审核机制。

这个边界是我认为 AI 数据产品落地时很重要的一点：AI 可以提高分析效率，但不能替代数据口径、计算逻辑和审核机制。

## 项目价值

InsightFlow 想展示的不是复杂工程能力，而是 AI 数据产品从问题理解、workflow 设计、数据分析到评测闭环的完整思路。

对我来说，这个项目的核心价值是把“智能问数”从一个概念拆成了可运行的模块，并且每个模块都有明确职责和测试依据。后续如果继续扩展，可以接入真实 LLM、真实数据库、更多 benchmark，以及更完整的归因分析逻辑。
