# 智能问数产品上线与迭代计划

## MVP 阶段

目标是验证核心 workflow 是否跑通，而不是覆盖所有数据场景。

范围：

1. 只覆盖一个业务域：本地生活或零售运营。
2. 只覆盖 6 个核心指标：GMV、订单量、用户数、客单价、高峰期订单、优惠券成本。
3. 只覆盖 4 类任务：下滑诊断、趋势分析、区域对比、影响分析。
4. 只使用只读数据源或 mock 数据源。
5. 必须展示结构化 intent、SQL 预览、指标口径和诊断依据。

交付物：

1. 可运行原型。
2. benchmark query 集合。
3. prompt 版本记录。
4. SQL 校验规则。
5. 人工评审表。

MVP 通过标准：

1. 标准问题可以稳定完成从提问到诊断的链路。
2. 非法字段和未知地区不会进入 SQL 执行。
3. 用户可以看到系统理解和数据依据。

## 内测阶段

目标是验证真实用户是否愿意使用，以及答案是否对业务分析有帮助。

内测用户：

1. 2 到 3 名城市运营。
2. 1 到 2 名数据分析师。
3. 1 名业务产品经理。

内测方式：

1. 每位用户提供最近一周真实问数问题。
2. 产品团队将问题脱敏后加入评测集。
3. 用户使用产品回答高频问题，并标记结果是否有用。
4. 数据分析师抽检 SQL 和口径。

关注指标：

1. 问题解析成功率。
2. 答案有用率。
3. 口径错误率。
4. 转人工率。
5. 用户愿意复用的场景。

内测结束后需要输出问题清单、失败样本、prompt 修复记录和下一轮上线建议。

## 灰度阶段

目标是在有限业务范围内验证稳定性和风险控制。

灰度策略：

1. 先开放给一个城市或一个业务线。
2. 每天限制查询量，避免 SQL 成本失控。
3. 高风险问题默认进入人工审核。
4. 每日复盘负反馈和拦截样本。
5. prompt 新版本只对小流量用户生效。

灰度期间必须监控：

1. SQL 校验拦截率。
2. schema 外输出比例。
3. 用户负反馈。
4. 权限拦截记录。
5. 平均响应时间和 P95 latency。

灰度通过标准：

1. 没有严重权限或口径事故。
2. 高频场景答案有用率稳定。
3. 失败样本可以被快速定位到 intent、SQL、数据或诊断环节。
4. 用户明确认可至少 2 个高频场景的效率提升。

## 全量阶段

目标是把能力从单一业务线扩展为稳定的数据分析入口。

全量前准备：

1. 指标字典和 schema 管理流程稳定。
2. 数据权限体系接入统一权限平台。
3. 评测样本库有固定负责人。
4. 负反馈到修复发布形成闭环。
5. 产品帮助文档说明能力边界。

全量后迭代方向：

1. 扩展更多指标和地区。
2. 支持更多时间范围表达。
3. 增加多轮追问，但每轮都保留结构化状态。
4. 增加图表解释和异常检测。
5. 接入更多业务线前先补充对应 benchmark。

## 反馈闭环

反馈闭环分为四类：

1. 用户反馈：有用、无用、口径错误、希望转人工。
2. 系统反馈：SQL 校验失败、schema 外输出、低置信度、超时。
3. 人工评审：分析师抽检、产品复盘、业务确认。
4. 数据反馈：指标字典变更、表结构变更、权限变更。

处理流程：

```text
反馈收集
  -> 问题分类
  -> 样本标注
  -> prompt 或规则修复
  -> 离线评测
  -> 灰度验证
  -> 发布或回滚
```

反馈优先级：

1. 数据权限和口径错误最高优先级。
2. 高频问题解析错误第二优先级。
3. 表达不清晰和体验问题第三优先级。
4. 低频探索类需求进入后续路线图。
