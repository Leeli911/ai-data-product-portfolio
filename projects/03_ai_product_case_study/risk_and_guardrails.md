# 风险边界与防护机制

## LLM Hallucination 风险

风险说明：

LLM 可能生成不存在的指标、字段、地区、任务类型或解释。例如把“优惠券成本”输出成不存在的 `coupon` 字段，或在没有数据证据时给出“补贴策略导致 GMV 下滑”的确定结论。

防护机制：

1. schema grounding：模型只能从合法指标、字段、地区和任务类型中选择。
2. JSON schema 校验：结构化输出必须符合固定字段和枚举值。
3. 低置信度降级：无法识别的问题进入澄清或人工审核。
4. 诊断证据绑定：每条解释必须能追溯到具体计算结果。
5. 线上 hallucination 监控：将 schema 外输出作为高优先级质量指标。

## SQL 误生成风险

风险说明：

NL2SQL 可能生成错误字段、错误聚合方式、错误时间范围，甚至产生高成本查询。如果直接执行，会带来错误结论、性能风险或权限风险。

防护机制：

1. 只读执行：只允许 `SELECT` 类查询，不允许写入、删除、更新和建表。
2. 字段白名单：SQL 中的表和字段必须来自已登记 schema。
3. 权限校验：根据用户身份过滤可访问业务线、城市、表和字段。
4. 查询成本限制：限制扫描范围、时间范围和返回行数。
5. SQL 预览和确认：MVP 阶段默认展示 SQL，让用户或分析师确认。
6. 执行前校验：SQL 必须通过语法、schema、权限和成本校验后才能执行。

## 指标口径错误风险

风险说明：

业务用户说“销售额”“流水”“GMV”时，可能指向不同口径。如果系统没有明确口径，就可能用错指标，导致错误决策。

防护机制：

1. 指标字典：每个指标维护名称、别名、公式、负责人和适用范围。
2. 口径卡片：答案中展示当前使用的指标定义和数据来源。
3. 口径冲突澄清：当一个表达命中多个口径时，先让用户选择。
4. 版本记录：指标口径变更需要记录生效时间和影响范围。
5. 分析师抽检：高频指标和高风险指标定期人工复核。

## 数据权限风险

风险说明：

智能问数降低了数据访问门槛，也可能让用户通过自然语言绕过权限边界，例如查询无权限城市、商户、用户分群或敏感字段。

防护机制：

1. 权限前置：在 SQL 生成和执行前校验用户可访问范围。
2. 敏感字段屏蔽：用户级、商户级、财务敏感字段默认不可查。
3. 结果脱敏：需要展示明细时按权限做聚合、脱敏或拒答。
4. 审计日志：记录用户问题、解析结果、SQL、执行结果和反馈。
5. 越权拒答：无法通过权限校验的问题直接拒答，不让模型尝试解释。

## 结果误读风险

风险说明：

用户可能把相关变化理解为因果结论，也可能忽略样本量、时间范围和数据延迟。AI 如果表达过于确定，会放大误读风险。

防护机制：

1. 结论分级：区分事实结果、相关性观察和待验证假设。
2. 置信度提示：对低样本量、数据延迟或异常波动标注提醒。
3. 不做过度归因：没有实验或因果模型支持时，不输出确定因果。
4. 展示计算依据：给出本周、上周、环比和主要拆解项。
5. 建议下一步验证：对影响分析给出可验证方向，而不是直接给业务决策。

## 人工审核机制

以下问题必须进入人工审核或澄清：

1. 涉及财务、预算、绩效和人员评价的问题。
2. 涉及敏感字段或跨权限范围的问题。
3. 指标口径存在冲突的问题。
4. SQL 校验失败但用户继续追问的问题。
5. 模型置信度低或多次解析不一致的问题。
6. 可能被业务方用于重大决策的问题。

人工审核不是为了降低自动化价值，而是为了把 AI 能力限定在可信边界内，让高频标准问题自动化，让高风险问题可控地进入人工流程。
