# 智能问数产品评测方案

## 评测目标

评测目标不是证明模型“能回答”，而是判断智能问数链路是否达到可灰度的稳定性。评测需要覆盖自然语言理解、schema grounding、SQL 生成、数据计算、诊断摘要和用户反馈。

核心问题：

1. 用户问题是否被正确解析成指标、地区、时间和任务类型。
2. 解析结果是否严格落在合法 schema 内。
3. SQL 是否可执行、只读、权限正确、成本可控。
4. 诊断是否基于真实计算结果，而不是模型主观编造。
5. 用户是否认为答案有用、清晰、可信。

## 离线评测指标

| 指标 | 定义 | 用途 |
|---|---|---|
| intent accuracy | metric、district、time_range、task 的整体命中率 | 判断问题理解是否正确 |
| metric accuracy | 指标识别准确率 | 发现指标同义词和复合指标问题 |
| district accuracy | 地区识别准确率 | 发现地名映射和多地区问题 |
| task accuracy | 任务类型识别准确率 | 判断趋势、对比、归因、影响分析是否区分清楚 |
| SQL validity | SQL 是否通过语法、字段、权限和只读校验 | 防止错误查询进入执行 |
| hallucination rate | 输出是否包含 schema 外指标、字段、地区或任务 | 衡量模型编造风险 |
| answer faithfulness | 诊断摘要是否完全基于返回数据 | 防止过度解释 |
| latency | 从提问到返回结果的耗时 | 判断体验是否可接受 |

离线 benchmark 需要包含：

1. 高频标准问题：如 GMV 下滑、订单趋势、地区对比。
2. 同义词问题：如销售额、流水、交易额映射到 GMV。
3. 复合指标问题：如晚高峰订单、优惠券成本。
4. 多地区和无地区问题：验证默认值和澄清策略。
5. 反例问题：包含不存在指标、越权字段和模糊口径。
6. 对抗问题：诱导模型编造字段、跳过 SQL 校验或直接给结论。

## 在线评测指标

1. 提问成功率：用户问题进入可执行链路的比例。
2. 澄清率：系统要求用户补充指标、地区或时间的比例。
3. 转人工率：低置信度、高风险或口径冲突问题进入人工审核的比例。
4. 答案有用率：用户点击“有帮助”的比例。
5. 口径纠错率：用户或分析师标记口径错误的比例。
6. SQL 拦截率：SQL 校验模块拦截非法字段、越权查询或高成本查询的比例。
7. 二次追问率：用户基于结果继续追问的比例，用于判断答案是否激发有效分析。
8. 负反馈恢复时间：问题从反馈进入修复、评测、发布的平均周期。

## 人工评审机制

人工评审由数据分析师、数据产品经理和业务代表共同完成。

评审对象：

1. 新增 prompt 版本。
2. 新增指标、地区或业务线。
3. 线上低分答案样本。
4. 被 SQL 校验拦截的问题。
5. 用户标记为“口径错误”或“结论不可信”的问题。

评审维度：

1. 解析是否正确。
2. SQL 是否符合数据口径。
3. 诊断是否只基于数据证据。
4. 表达是否适合业务用户理解。
5. 是否需要澄清、拒答或人工介入。

评审结论进入样本库，作为下一轮 prompt 和 benchmark 迭代依据。

## Prompt 版本迭代机制

每个 prompt 版本必须有版本号、变更说明、适用范围和评测结果。

迭代流程：

```text
收集问题样本
  -> 标注标准答案
  -> 修改 prompt 或 schema 配置
  -> 运行离线 benchmark
  -> 人工抽检失败样本
  -> 小流量灰度
  -> 线上反馈回收
  -> 决定保留、回滚或继续迭代
```

版本发布规则：

1. 不允许只凭单个 demo case 发布新 prompt。
2. 新版本必须和上一版本做横向对比。
3. 如果 hallucination rate 上升，新版本不得进入灰度。
4. 如果 SQL validity 下降，新版本必须回滚或修复。
5. 如果低频复杂问题提升但高频问题下降，需要拆分适用场景。

## 上线门槛

MVP 内测门槛：

1. 核心 benchmark intent overall accuracy 不低于 0.85。
2. SQL validity 达到 1.00。
3. 核心 benchmark hallucination rate 达到 0。
4. answer faithfulness 人工抽检通过率不低于 0.90。
5. P95 latency 满足业务可接受范围。

灰度门槛：

1. 连续两轮离线评测无重大回归。
2. 内测用户答案有用率达到 0.70 以上。
3. 口径错误和越权查询有明确拦截记录。
4. 高风险问题可以稳定进入澄清、拒答或人工审核。

全量门槛：

1. 灰度期间无严重数据权限事故。
2. 线上负反馈可追踪、可复盘、可修复。
3. 业务方认可高频问题响应效率提升。
4. 数据团队可以维护指标口径、schema 和评测样本库。
