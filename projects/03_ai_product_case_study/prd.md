# 智能问数与自动归因分析产品 PRD

## 背景

本地生活、外卖和零售业务的经营分析高度依赖数据。运营、产品和管理层每天都会关注 GMV、订单量、用户数、客单价、高峰期订单和优惠券成本等指标，但多数业务用户并不熟悉 SQL，也不一定清楚指标口径和数据表结构。

当前常见流程是：业务方提出问题，分析师确认口径、写 SQL、取数、计算环比，再整理原因解释。这个流程能保证准确性，但响应链路较长，重复问题多，分析师也容易被基础取数需求占用。

智能问数与自动归因产品希望把高频、标准化的经营分析问题前置自动化，让业务用户可以用自然语言提问，系统完成问题理解、取数逻辑生成、数据校验和结构化诊断。产品设计原则是：AI 负责理解和组织，数据计算、SQL 校验、权限控制和口径治理必须由确定性系统完成。

## 目标用户

1. 城市运营：关注城市、商圈、区域维度的 GMV、订单和用户变化，需要快速定位异常原因。
2. 产品经理：关注功能改版、补贴策略和活动对核心指标的影响，需要快速验证方向。
3. 数据分析师：希望减少重复取数和基础解释，把精力放在复杂专题分析上。
4. 业务管理层：需要快速获得经营概览、异常摘要和可追溯的数据依据。

## 用户痛点

1. 不会写 SQL：业务问题是自然语言表达，数据平台需要结构化查询条件。
2. 口径不统一：同一个指标可能存在多个口径，业务用户很难判断应该使用哪一个。
3. 归因慢：指标变化后需要反复查 GMV、订单、用户、客单价、优惠券成本等拆解项。
4. 分析重复：大量问题集中在固定指标、固定地区、固定时间范围，分析师重复劳动多。
5. AI 答案不可控：如果没有 schema、SQL 和口径校验，模型容易编造字段或给出过度解释。

## 产品目标

1. 降低高频经营分析的提问门槛，让业务用户可以用自然语言完成基础问数。
2. 缩短标准化问题的响应时间，把分钟级或小时级分析压缩到秒级或分钟级。
3. 保证关键数据结论可追溯，用户可以看到解析结果、SQL 预览、指标口径和数据来源。
4. 建立 AI 输出质量评估机制，用离线评测和线上反馈驱动 prompt 与 workflow 迭代。
5. 明确能力边界，避免把不确定推断包装成确定结论。

## 核心场景

1. 指标下滑诊断：用户提问“为什么北京朝阳区本周 GMV 下滑”，系统输出本周、上周、环比变化和主要影响因素。
2. 趋势分析：用户提问“海淀区最近订单趋势如何”，系统返回时间序列摘要和异常节点。
3. 区域对比：用户提问“哪个地区客单价下降最多”，系统按地区聚合并返回排序结果。
4. 影响分析：用户提问“优惠券成本上升是否影响 GMV”，系统展示优惠券成本和 GMV 的变化关系，并标注不能直接证明因果。
5. 指标解释：用户点击指标名后，系统展示口径、计算公式、更新时间和适用业务线。

## 核心功能

1. 智能问数输入：支持中文自然语言问题，识别指标、地区、时间范围和任务类型。
2. 结构化 intent 展示：在执行前展示系统理解结果，允许用户确认或修改。
3. SQL 预览：展示只读 SQL，标注使用的表、字段、筛选条件和聚合逻辑。
4. 指标口径卡片：展示指标定义、计算公式、数据负责人和适用范围。
5. 自动归因摘要：基于确定性计算输出环比变化、拆解指标和主要变化因素。
6. 置信度提示：对低置信度解析、跨口径问题和高风险解释给出提示或转人工。
7. 反馈入口：用户可以标记答案是否有用、口径是否正确、是否需要分析师介入。

## AI Workflow

```text
用户自然语言提问
  -> query understanding：识别问题意图和分析任务
  -> schema grounding：限定可用指标、字段、地区和任务类型
  -> intent validation：校验结构化输出是否完整、合法、可执行
  -> SQL generation：生成只读 SQL 或查询计划
  -> SQL validation：校验表、字段、权限、聚合方式和查询成本
  -> metric calculation：由数据系统执行确定性计算
  -> diagnosis generation：基于计算结果生成解释摘要
  -> answer review：展示 SQL、口径、置信度和反馈入口
```

关键原则：

1. LLM 不直接计算 GMV、环比和排序。
2. LLM 不直接执行 SQL。
3. 所有 SQL 必须通过白名单、权限和成本校验。
4. 所有诊断必须基于已返回的数据证据。
5. 低置信度或高风险问题必须降级为澄清、拒答或人工审核。

## MVP 范围

1. 支持 6 个核心指标：GMV、订单量、用户数、客单价、高峰期订单、优惠券成本。
2. 支持 4 类任务：下滑诊断、趋势分析、区域对比、影响分析。
3. 支持 4 个示例地区：朝阳、海淀、丰台、浦东。
4. 支持本周、上周、最近 7 天、最近 14 天等基础时间表达。
5. 支持 SQL 预览、指标口径展示和结果反馈。
6. 先接入离线 mock 数据或只读数据集，不接入生产写权限。

## 非目标范围

1. 不支持用户自由查询所有数据库和所有字段。
2. 不支持未经校验的自动 SQL 执行。
3. 不支持让 LLM 直接给出财务、预算或绩效决策建议。
4. 不支持复杂因果推断，影响分析只展示相关变化和可疑因素。
5. 不在 MVP 阶段覆盖多轮深度专题分析和跨业务线复杂口径。

## 成功指标

1. Intent overall accuracy 达到 MVP 上线门槛。
2. SQL validity 达到 100%，不允许生成 schema 外字段进入执行。
3. Hallucination rate 在离线核心集上为 0。
4. 高频问题平均响应时间明显低于人工取数链路。
5. 业务用户答案有用率达到内测门槛。
6. 分析师重复取数需求量下降。
7. 高风险问题转人工率和误答率可被监控。
